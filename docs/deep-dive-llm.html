Deep Dive: LLM Prompts

Every sentence in every prompt, dissected. Why each rule exists, what breaks if you remove it, and the prompt engineering principles behind the design.

⚡ The Haiku Classifier Prompt

Located in: claudeService.ts → isRecipeVideo()

# System prompt: "You classify social media videos. Reply with only YES or NO." # User message (template): Video title: [metadata.title] Creator: [metadata.uploader] Caption (first 500 chars): [metadata.description.slice(0, 500) || "(none)"] Does this video teach the viewer how to cook or make a recipe at home? Reply with only YES or NO. # Parameters: max_tokens: 5 # YES = 1 token. Buffer for "YES," edge case
❌ If system prompt removed
Haiku defaults to a helpful assistant role and might respond with "Yes, this appears to be a cooking video! The creator shares..." — breaking the single-token YES/NO assumption in the parser.
❌ If "Reply with only YES or NO" removed from user message
The model might respond "Based on the caption, I believe this is indeed a recipe video." — startsWith("YES") still works, but it wastes tokens and increases cost.
❌ If caption truncated to 100 chars instead of 500
Some recipe captions start with an emoji or a tagline before the ingredient list. 100 chars might not reach the recipe content. 500 chars reliably captures the opening structure.
✓ Why "teaches the viewer how to...at home"
Explicitly excludes restaurant tours (where food is shown but not made at home), mukbangs (eating without instructions), and cooking competitions. The "at home" qualifier is critical.
🏛️ The System Prompt — Line by Line

Located in: promptBuilder.ts → SYSTEM_PROMPT

SYSTEM_PROMPT — annotated sentence by sentence
You are a culinary AI assistant that analyses social media video metadata. Role assignment. "Culinary AI" anchors Claude in food/cooking domain knowledge. "Social media video metadata" sets the context — Claude should reason about TikTok/Instagram captions, not assume a formal recipe card format. Without this, Claude might apply kitchen-book recipe conventions to informal social captions.
You receive video titles, descriptions/captions, and sometimes a thumbnail image. Input declaration. Tells Claude what data to expect and use. "Sometimes" prepares it for absent thumbnail — so it doesn't get confused when no image is provided.
Your first job is to decide: is this actually a recipe or cooking video? Task framing. Making this "first job" ensures Claude checks content validity before extraction. Without this, Claude might extract partial "recipe data" from a restaurant tour video.
CRITICAL RULE — set "is_recipe": false (and leave all other fields null/empty) if the video is NOT a recipe or cooking instruction. Primary guard rule. "CRITICAL RULE" signals high importance to Claude. The all-caps is an intentional emphasis signal in prompt engineering. The instruction to leave fields null/empty on non-recipes prevents Claude from hallucinating partial recipe data for irrelevant videos. Note: the current schema doesn't actually include is_recipe — Claude has been conditioned to throw this away and just extract (or not extract) based on this rule.
Examples of non-recipe content: travel vlogs, fitness/workout, comedy sketches, product reviews, restaurant visits without a recipe, lifestyle content, music videos, etc. Negative examples. Explicitly listing exclusions dramatically improves precision. Without examples, Claude might extract "recipes" from: a food tour ("we tried their famous pasta"), a mukbang, or a cooking show advertisement. Each example covers a common edge case in social media content.
Only set "is_recipe": true if the video teaches the viewer how to make or cook something at home. Positive definition. "Teaches" + "at home" are the two key criteria. A restaurant visit video is still cooking content but doesn't teach viewers — it's excluded. "At home" excludes professional cooking competitions where the technique isn't reproducible.
When is_recipe is true, extract all recipe information into precise JSON. Additional rules: Context switch signal. "When is_recipe is true" makes the extraction rules conditional, reinforcing that they only apply after the validity check.
- Be literal with measurements — never invent quantities not mentioned in the source Anti-hallucination rule #1. Without this, Claude might infer "2 cups flour" from context when the video only says "some flour." Food quantities matter — an incorrect measurement ruins the dish. "Literal" and "never invent" are strong directives.
- If a field is unclear or missing, use null rather than guessing Anti-hallucination rule #2. Allows Claude to express uncertainty explicitly via null rather than inventing a plausible value. This is critical for quantities — a guessed "2 tbsp" when the video is unclear is worse than null. The app handles null gracefully; a hallucinated measurement doesn't degrade gracefully on the stove.
- Combine duplicate ingredients (e.g. "2 cloves garlic" mentioned twice → list once) Deduplication rule. Social media captions often mention the same ingredient multiple times (in the ingredient list and in the instructions). Without this rule, the ingredient list would have duplicates. The example makes the expected behavior unambiguous.
- Steps should be complete sentences, one action per step Step format rule. Prevents fragmented steps like "Salt" or "High heat." Complete sentences are more useful: "Season with salt and bring to a boil." "One action per step" prevents multi-action steps that are hard to follow while cooking.
- Difficulty: "easy" (< 30 min, few ingredients), "medium" (30–60 min), "hard" (> 60 min or complex techniques) Difficulty calibration. Provides explicit criteria for the difficulty enum — time-based and complexity-based. Without this, Claude's difficulty assessments would be inconsistent across recipes. Matching the schema's "easy"/"medium"/"hard" enum values exactly.
- Respond ONLY with a valid JSON object wrapped in ```json ... ``` — no other text Output format constraint. The most critical rule for parseability. Claude naturally adds preamble ("Here is the extracted recipe:") and postamble ("Let me know if you need anything else!"). These would break JSON.parse(). The code fence format allows parseClaudeResponse() to reliably extract just the JSON.
📋 The Recipe Schema — Field by Field

Located in: promptBuilder.ts → RECIPE_SCHEMA

Design choice: String schema vs JSON Schema
RECIPE_SCHEMA is a plain string template, not a formal JSON Schema or Zod definition. Claude understands natural language type annotations like "integer | null" and "float 0.0–1.0" better than formal schema syntax like {"type": ["integer", "null"]}. The string schema also serves as a few-shot example — showing Claude the exact format expected with example values inline. Using Anthropic's tool_use feature would give stronger output guarantees but requires restructuring the prompt as a tool definition.
FieldType in SchemaWhy this type / Why nullable
title string Clean recipe name — "not the creator's name or channel" is explicit because many TikTok captions start with the creator's name. Without this instruction, Claude might return "Chef Julia's Pasta" instead of "Garlic Butter Pasta".
cook_time_minutes integer | null Nullable because many recipe videos don't mention total cook time explicitly. Integer (not float) to avoid "20.5 minutes" which is unusual in recipe contexts.
prep_time_minutes integer | null Nullable — even less commonly mentioned than cook time in short-form videos. The distinction between prep and cook time matters for planning.
servings integer | null Nullable — often omitted in TikTok recipes. Integer is appropriate: "2.5 servings" doesn't make sense in practice.
difficulty "easy"|"medium"|"hard"|null Closed enum — only these exact strings are accepted. The SYSTEM_PROMPT provides time-based criteria so difficulty is objective, not subjective. Null when the recipe can't be classified.
tags string[] — e.g. "salmon", "dinner", "30-minute" Non-nullable array — always returns at least []. The examples guide Claude to return ingredient-based, meal-type, and time-based tags. Used for iOS category filtering.
ingredients[].name string (required) Always present — it's the primary identifier of an ingredient. The normalization code falls back to "Unknown ingredient" if Claude somehow omits it.
ingredients[].quantity string | null — e.g. "2", "1/2", "a pinch" String, not number — because quantities are often fractions ("1/2"), ranges ("2-3"), or qualitative ("a handful"). Storing as string preserves the original phrasing.
ingredients[].unit string | null — e.g. "cups", "tbsp", "g" Nullable — "2 garlic cloves" has a unit, "a pinch of salt" doesn't have a discrete unit. Stored separately from quantity so the iOS app can display "500 g butter" with proper formatting.
ingredients[].notes string | null — e.g. "finely chopped", "room temperature" Nullable — preparation notes are optional but valuable. Storing separately from name keeps the ingredient name clean ("garlic" not "finely chopped garlic") while preserving the preparation detail.
steps[].step_number integer starting at 1 "starting at 1" is explicit — prevents 0-based numbering. The normalization code uses array index + 1 as fallback if Claude returns 0-based.
steps[].instruction string — "complete actionable sentence" "Actionable" means imperative voice ("Add the butter and stir") not descriptive ("The butter is added"). "Complete sentence" prevents fragments.
steps[].duration_minutes integer | null Nullable — most steps don't have explicit durations. When present (e.g. "simmer for 10 minutes"), captured for the iOS timer feature.
confidence float 0.0–1.0 Claude's self-assessment of extraction quality. High confidence (>0.85): clear caption with all details. Low confidence (<0.5): sparse caption, video relies on visual demonstration. The app could display a warning for low-confidence extractions. Clamped to [0,1] in normalization.
💬 The User Message Structure

Located in: promptBuilder.ts → buildExtractionPrompt()

role: "user" — content: ContentBlockParam[]
① text block
Analyse this social media video and extract the recipe if it is one. Platform: {metadata.platform} ← "tiktok" or "instagram" Video Title: {metadata.title} Creator: {metadata.uploader} Duration: {metadata.duration}s ← omitted if null Caption/Description: --- {captionText (up to 3000 chars)} --- Extract into this exact JSON schema: {RECIPE_SCHEMA}
② image block (only if thumbnail available)
{type:"image", source:{type:"base64", media_type:"image/jpeg", data: base64Data}}

The base64 data URI prefix (data:image/jpeg;base64,) is stripped before sending — Anthropic requires raw base64 only.
③ text block (only if thumbnail present)
"The image above is a thumbnail from the video. Use it to help identify the dish if the caption is sparse."

This guidance text prevents Claude from ignoring the image and tells it when/how to use it.
Why "---" around caption?
The triple-dash separators clearly delineate where the caption starts and ends. Without them, Claude might conflate long captions with the schema instructions below. This is a common technique for separating injected user content from instructions.
Why include Platform?
TikTok captions are more informal (emoji-heavy, bullet-point ingredients). Instagram captions are often longer with paragraph-format recipes. Providing the platform helps Claude calibrate its parsing heuristic for the caption structure.
Why caption limited to 3000 chars?
3000 chars ≈ 750 tokens. Most recipe captions are under 500 chars. 3000 is generous enough for long recipe posts while keeping input token cost controlled. Claude's recipe extraction quality doesn't meaningfully improve beyond 3000 chars of caption.
🖼️ Multimodal Design: Text + Image
When does the thumbnail actually help?
The thumbnail helps most when: (1) the caption is sparse ("Make this! #recipe") with no ingredient list, (2) the dish name isn't obvious from the title, (3) the presentation format (plated vs. unplated) suggests complexity. The thumbnail helps least when the caption already has a full ingredient list — Claude's text comprehension is more reliable than vision for structured data. Adding the guidance text in block ③ ("use it to help identify the dish if the caption is sparse") conditions Claude to reference the image when text is insufficient, rather than on every call.

Multimodal message structure (Anthropic API format):

// What buildExtractionPrompt() returns: { "system": "You are a culinary AI assistant...", "messages": [ { "role": "user", "content": [ { "type": "text", "text": "Analyse this social media video..." }, { "type": "image", "source": { "type": "base64", "media_type": "image/jpeg", "data": "/9j/4AAQSkZJRgAB..." // raw base64, NO data: prefix } }, { "type": "text", "text": "The image above is a thumbnail..." } ] } ] }
⚠️ Common mistake: data URI prefix in base64
Browser environments produce base64 as a data URI: "data:image/jpeg;base64,/9j/4AAQ...". Anthropic's API requires ONLY the base64 data without the prefix: "/9j/4AAQ...". The code strips it with: thumbnailBase64.replace(/^data:image\/\w+;base64,/, ""). Sending the full data URI would cause a 400 error from the API ("invalid base64 data").
🧠 Prompt Engineering Principles Applied
1. Role Assignment
"You are a culinary AI assistant" sets the persona before any task description. Models perform better when given a specific role — it activates relevant domain knowledge and establishes behavioral expectations.
2. Negative Examples
Listing what the model should NOT do ("travel vlogs, fitness, comedy") is as important as listing what it should do. Negative constraints are often more effective than positive definitions at the boundary cases.
3. Format Constraint
"Respond ONLY with a valid JSON object wrapped in ```json...```" is the output format constraint. Explicit format constraints are the only reliable way to make an LLM produce machine-parseable output. The "no other text" clause is essential.
4. Calibrated Examples in Schema
The RECIPE_SCHEMA includes example values: "e.g. 'salmon', 'dinner', '30-minute'" for tags. Examples in the schema teach the expected vocabulary and granularity — without them, Claude might return overly broad tags ("food") or overly specific ones ("garlic-and-herb-butter-pasta").
5. Uncertainty Handling
"If a field is unclear or missing, use null rather than guessing" tells the model how to handle uncertainty. Without this, LLMs default to plausible-sounding fabrications. Explicit null permission reduces hallucination for missing data.
6. Self-Assessment Signal
The confidence field asks Claude to report its own extraction confidence. This is useful because Claude's uncertainty correlates with actual extraction quality — low confidence reliably flags cases where the caption was sparse or ambiguous. It's free signal from the model.
What would improve this further: Anthropic Tool Use
The current approach uses a string schema and parses Claude's text response. A more robust alternative: use Anthropic's tool_use feature where the schema is defined as a tool definition and Claude is forced to "call" it with valid JSON arguments. This would: (1) guarantee JSON schema compliance — no parsing needed; (2) eliminate the regex step entirely; (3) catch type errors (string where integer expected) at the API level. The migration would require restructuring the prompt as a tool definition, but the system prompt's instruction text would remain the same.
