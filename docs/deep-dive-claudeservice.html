Deep Dive: claudeService.ts

The AI heart of RecipeWizard. Two Claude calls, one error class, one parser, and careful type normalization. Every decision explained.

🎯 What This Module Does

claudeService.ts is responsible for one thing: given video metadata + an optional thumbnail, return a structured recipe — or throw if the video isn't a recipe.

It does this in two stages:

  1. Stage 1 (guard) — ask Haiku "is this a recipe?" with a 5-token budget. If NO, throw NotARecipeError and skip Stage 2 entirely.
  2. Stage 2 (extraction) — ask Sonnet to extract the full structured recipe from the caption + thumbnail. Parse, validate, normalize, and return.

The module exports three things: NotARecipeError, parseClaudeResponse (also used in tests), and extractRecipeFromVideo.

⚡ Stage 1: isRecipeVideo()
What this function does
A lightweight binary classifier. Sends the video title, creator name, and first 500 characters of the caption to claude-haiku — the cheapest Claude model — and asks a single YES/NO question. If the answer is NO, the entire extraction pipeline is aborted and the route returns a 422 error to the iOS app. This is the cost-control gatekeeper.

39–44const prompt = `Video title: ${metadata.title} Creator: ${metadata.uploader} Caption (first 500 chars): ${metadata.description.slice(0, 500) || "(none)"} Does this video teach the viewer how to cook...? Reply with only YES or NO.`The question "teaches the viewer how to cook or make a recipe at home" is precisely worded. It rejects: restaurant visits, mukbangs, food reviews, cooking competitions. It accepts: step-by-step recipe tutorials. The "(none)" fallback handles videos with no caption.

Reply with only YES or NO.", messages: [{ role: "user", content: prompt }], });max_tokens:5 is the critical parameter. YES costs 1 token, NO costs 1 token. 5 gives some buffer if the model prefaces with "YES," but it's still extremely cheap. The system prompt reinforces the binary reply constraint.
53–58const answer = response.content[0].type === "text" ? response.content[0].text.trim().toUpperCase() : "NO"; console.log("[claude] isRecipeVideo answer:", answer); return answer.startsWith("YES");response.content[0].type === "text" check: Anthropic SDK returns a union type — content blocks can be "text" or "tool_use". Important: Why Haiku instead of a simple rule-based check?
One might think: "just check if the caption contains 'cups', 'tbsp', or 'recipe'". This fails for: (1) recipes with no explicit measurements ("add garlic until fragrant"); (2) non-recipes that mention food quantities ("this restaurant pours 3 cups of oil on every dish"); (3) foreign-language captions. Haiku understands context and intent — it handles all these cases in a single call for fractions of a cent, better than any regex heuristic.
🧠 Stage 2: extractRecipeFromVideo()
What this function does
The main exported function. Calls isRecipeVideo() as a guard, then calls buildExtractionPrompt() (from promptBuilder.ts) to construct the multimodal API payload, then calls claude-sonnet-4-6 with the full prompt. After Claude responds, calls parseClaudeResponse() to extract the JSON, then normalizes every field with null-coalescing and type validation before returning.
61export async function extractRecipeFromVideo( metadata: VideoMetadata, thumbnailBase64: string | null ): Promise<Omit<RecipeResponse, "source_url" | "platform" | ...>>Return type is Omit<RecipeResponse, "source_url" | "platform" | "thumbnail_url" | ...> — it explicitly excludes fields that claudeService can't know (they're added by the route handler from the request context). This is precise TypeScript design — narrow to exactly what this function can provide.

🔬 Response Parsing: parseClaudeResponse()
What this function does
Extracts JSON from Claude's text response. Two-level strategy: (1) regex for the expected ```json...``` code fence, (2) direct JSON.parse fallback if the fence is absent. Exported separately so it can be tested in isolation.
27const match = text.match(/```json\s*([\s\S]*?)```/);Regex anatomy: ``` three literal backticks, json literal "json", \s* zero or more whitespace chars (handles newline after ```json), ([\s\S]*?) capture group matching ANY character including newlines LAZILY, ``` closing backticks. [\s\S] is "whitespace or non-whitespace" = "everything" since . skips \n.
28–35if (!match) { try { return JSON.parse(text); } catch { throw new Error("Claude response did not contain valid JSON"); } } return JSON.parse(match[1]);match[0] = entire regex match including backtick fences. match[1] = capture group 1 = just the JSON text inside the fences. Fallback path handles edge cases where Claude ignores the fence instruction. Throws a meaningful error if both strategies fail.
Concept: Lazy vs greedy regex quantifiers
[\s\S]* is greedy — it matches as much as possible. [\s\S]*? is lazy — it matches as little as possible. The difference matters when there are multiple code fences in the response: greedy would span from the first ```json to the LAST ```, potentially including unrelated text. Lazy stops at the FIRST ``` after the opening fence. For Claude's recipe response which always has exactly one JSON block, both would work — but lazy is the correct and safer choice.
🔧 Response Normalization

After parsing, Claude's raw extraction needs normalization because LLMs can omit fields or use unexpected types. Every field is guarded:

90–95const ingredients = (extracted.ingredients ?? []).map((ing) => ({ name: ing.name ?? "Unknown ingredient", quantity: ing.quantity ?? null, unit: ing.unit ?? null, notes: ing.notes ?? null, }));?? [] guards against Claude omitting ingredients entirely (would return null or undefined). Each field uses ?? null except name which falls back to "Unknown ingredient" — a required DB field. This ensures every ingredient object always has exactly 4 keys with valid types.
97–101const steps = (extracted.steps ?? []).map((step, i) => ({ step_number: step.step_number ?? i + 1, instruction: step.instruction ?? "", duration_minutes: step.duration_minutes ?? null, }));step_number fallback to i+1 (array index + 1). If Claude omits step_number or uses 0-based numbering, we still get a valid 1-based sequence. Empty string fallback for instruction prevents DB NOT NULL constraint failures.
103–105const confidence = typeof extracted.confidence === "number" ? Math.max(0, Math.min(1, extracted.confidence)) : 0.5;typeof check prevents Math.max(0, Math.min(1, NaN)) = NaN. Clamp prevents values outside [0,1] from failing the PostgreSQL Float column constraint or confusing the iOS confidence display. 0.5 neutral default for absent confidence.
107–118return { title: extracted.title ?? metadata.title ?? "Untitled Recipe", description: metadata.description.slice(0, 500) || null, cook_time_minutes: extracted.cook_time_minutes ?? null, ... };title has a three-level fallback: extracted title → video title from yt-dlp → "Untitled Recipe". description uses the raw caption (not Claude's extraction) — more reliable source. || null converts empty string to null (a semantic "no description").
Good practice: Defensive normalization at the API boundary
The RawExtraction interface uses Partial types for all fields (ingredients?: Partial<Ingredient>[]). This accurately reflects that Claude might return any subset of the schema. The normalization step converts this uncertain input into a guaranteed-valid RecipeResponse. This is the "parse, don't validate" principle — transform input into a fully valid type once at the boundary rather than sprinkling type checks throughout the codebase.
❌ Error Handling Strategy
NotARecipeError
Thrown by: isRecipeVideo() → NO answer
Caught by: route handler
HTTP response: 422 Unprocessable Entity
iOS behavior: "not a recipe" alert shown to user
Parse Error
Thrown by: parseClaudeResponse()
Caught by: extractRecipeFromVideo() try/catch
HTTP response: 500 with detail message
Cause: Claude returned non-JSON text (very rare)
Anthropic API Error
Thrown by: client.messages.create()
Propagates up to route handler
HTTP response: 500
Cause: Invalid API key, rate limit, network issue
Image Service Error
Swallowed by: downloadAndEncodeImage()
Returns: null
Effect: recipe extraction continues without image
Design: thumbnail is always optional
Design: Error-as-value vs throw
Notice two different error strategies: imageService.downloadAndEncodeImage() returns null on failure (error-as-value); claudeService.extractRecipeFromVideo() throws on failure (exception). The difference is intentional: a failed thumbnail is a degraded but valid result — the recipe can still be extracted. A failed Claude call is a fatal failure — there's no recipe to return. Use error-as-value for recoverable degraded states; use exceptions for unrecoverable failures.
