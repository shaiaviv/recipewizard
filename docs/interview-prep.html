Interview Prep

Fully written answers to every likely interview question about this project. Read them out loud a few times.

🎤 "Walk me through the project"

RecipeWizard is a full-stack iOS application that solves a specific problem: when you see a cooking recipe on TikTok or Instagram, there's no easy way to save the actual recipe with ingredients and steps — you either screenshot the video, try to remember the caption, or lose it. RecipeWizard lets you share the video directly to the app, and it automatically extracts the full structured recipe using AI.

The backend is a Node.js/TypeScript Express API deployed on Railway. When the iOS app shares a TikTok or Instagram URL, the backend runs a four-stage pipeline. First, it spawns yt-dlp — a command-line tool — to fetch the video's metadata without downloading any video. That gives us the title, caption, and thumbnail URL. Second, it downloads and resizes the thumbnail using sharp, a C++ image processing library, to prepare it for the AI vision call.

The third stage is the AI extraction, which I designed as a two-stage pipeline for cost efficiency. I first send the title and a snippet of the caption to Claude Haiku — the cheapest and fastest Claude model — with a simple YES/NO question: "Does this video teach the viewer how to cook?" This pre-filter costs fractions of a cent and blocks non-recipe content — travel vlogs, mukbangs, comedy sketches — before touching the expensive model. If Haiku says YES, I call Claude Sonnet 4.6 with the full caption plus the thumbnail image, using a carefully engineered system prompt that defines the exact JSON schema I want back.

After extraction, there's an optional fourth stage: Gemini 2.5 Flash generates a Studio Ghibli-style watercolor illustration of the dish, which replaces the thumbnail in the app for a more consistent visual experience. Finally, everything is persisted to PostgreSQL via Prisma ORM, and the response is sent back to the iOS client.

Authentication uses Google Sign-In. The iOS app gets a Google ID token, sends it to the backend, which verifies it with Google's official library, upserts the user in the database, and returns a JWT with a 30-day expiry. The JWT is stored in a shared App Group UserDefaults so the Share Extension can include it in API requests even though it's a separate iOS process.

🔧 Technology Questions
What is yt-dlp and why did you use it?

yt-dlp is an open-source command-line tool that can extract metadata and media from over 1,000 video platforms. It's a maintained fork of youtube-dl with better support for TikTok, Instagram, and other modern platforms.

I used it for one specific feature: --dump-json — which tells yt-dlp to output the video's metadata as JSON and exit immediately, without downloading a single byte of video data. That gives me the title, description, thumbnail URL, uploader, duration, and platform detection in about 1–2 seconds.

The alternatives were: (1) scraping the HTML of TikTok/Instagram pages directly — fragile, breaks when they change their markup; (2) using official APIs — TikTok and Instagram have very restricted API access that doesn't include recipe data from user videos; (3) downloading the video and extracting text from frames — extremely expensive and slow. yt-dlp was the only practical path.

Why two AI models — Haiku and Sonnet?

It's a cost-optimization pattern I call "classifier gating." The problem is that users often share non-recipe content — they might accidentally share a fitness video, a travel vlog, or a restaurant review. Running the full claude-sonnet-4-6 extraction on those would waste money and return a confusing error.

So I added a pre-filter: claude-haiku-4-5 with max_tokens: 5 and a YES/NO question. Haiku is roughly 10-15x cheaper than Sonnet per token, and a 5-token YES/NO response costs fractions of a cent. For a typical user who shares 50% non-recipe content, this halves the AI cost without affecting accuracy at all — the actual recipe extraction only runs on confirmed cooking videos.

The tradeoff is two sequential API calls instead of one. The latency overhead is small because Haiku responds in under a second.

What is Prisma and why use an ORM instead of raw SQL?

Prisma is a TypeScript ORM that generates a fully-typed client from a schema.prisma definition. The killer feature is that when I define a model in the schema, TypeScript automatically knows the exact shape of every query result — no type drift between the database and the application code.

For example, when I call prisma.recipe.findMany({include: {ingredients: true}}), TypeScript knows the return type includes a Recipe with an ingredients: Ingredient[] array — because Prisma generated that type from the schema.

I also used Prisma's nested write syntax to insert a Recipe + its Ingredients + its Steps in a single atomic transaction:

prisma.recipe.create({ data: { title, userId, ingredients: { create: recipe.ingredients }, steps: { create: recipe.steps }, } })

This is cleaner and safer than writing three separate INSERT statements. Raw SQL would have required manual parameterization to prevent SQL injection and manual type mapping.

Why sharp for image processing?

sharp is a Node.js binding for libvips, a C++ image processing library. It's 100–1000x faster than pure JavaScript image libraries like Jimp because the heavy lifting happens in native code. For an API endpoint that processes one image per request, this isn't a bottleneck — but sharp also has a much better API for what I needed: resize, convert format, control quality, all in a single method chain.

The specific operation: sharp(buffer).resize({width: 1024, withoutEnlargement: true}).jpeg({quality: 85}).toBuffer() — takes a possibly-large WebP or PNG thumbnail, downsizes it if wider than 1024px (without upscaling small images), converts to JPEG at 85% quality, and returns a Buffer. This keeps the base64 payload to Claude under ~200 KB, which is well within Anthropic's limits and reduces API cost.

How does Google OAuth + JWT authentication work?

There are two parts. First, the iOS app uses Google Sign-In SDK, which handles the OAuth flow natively — the user sees the Google consent sheet, Google issues a short-lived ID token (a signed JWT from Google). The app sends this token to my backend.

On the backend, I use Google's official google-auth-library to verify the token. The library fetches Google's public keys from their JWKS endpoint, verifies the token's RS256 cryptographic signature, and extracts the claims — including the user's stable Google ID (sub), email, name, and avatar URL. I trust this library completely — DIY verification would be a security mistake.

Once verified, I do a Prisma upsert — create the user if they're new, update their profile if they already exist. Then I sign my own JWT with a 30-day expiry using jsonwebtoken. The iOS app stores this JWT in UserDefaults (inside the shared App Group so the Share Extension can use it too) and sends it as a Bearer token on every subsequent request.

The backend's requireAuth middleware then calls jwt.verify(token, JWT_SECRET) on every protected route to validate the token.

What is SwiftData and how did you use it?

SwiftData is Apple's persistence framework introduced in iOS 17 — it's the modern replacement for Core Data, with much cleaner Swift-native syntax. You define your models with @Model, and it handles the SQLite storage, migrations, and SwiftUI integration automatically.

In RecipeWizard, the three SwiftData models are Recipe, Ingredient, and RecipeStep. They live in the main app target and use @Relationship for the associations. SwiftUI views use @Query to fetch recipes with automatic sort and filter, and the view updates automatically when data changes.

One important constraint: CloudKit sync is currently disabled. CloudKit requires that all model properties are optional or have defaults, and arrays must be stored as JSON-encoded Data (not native Swift arrays) because CloudKit doesn't support arrays natively. I've documented this in the code but left it disabled pending a paid Apple Developer account.

🏗️ Architecture Questions
Why is the DB save non-blocking (fire-and-forget)?

The recipe extraction pipeline already takes 3–8 seconds due to yt-dlp + two Claude calls + Gemini. Adding a DB write (which takes 50–200ms) to the critical path would be noticeable latency for zero benefit from the user's perspective — they already have the recipe data in the response.

The implementation is: saveRecipeForUser(userId, payload).catch(err => console.error(...)) — no await, with a catch handler that logs failures without crashing. The HTTP response fires immediately after res.json(responsePayload).

The risk is that if the Node process crashes in the milliseconds between res.json() and the DB write completing, the recipe is lost from the DB (though the iOS app already has it locally in SwiftData). For the current scale and use case, this is an acceptable tradeoff. For a payment system, this pattern would be unacceptable — every state change would need durable recording before responding.

How does the Share Extension work? Why can't it use SwiftData directly?

The Share Extension is a separate iOS process with a strict 120MB memory limit — SwiftData's full persistence stack doesn't fit within that budget. Also, because it's a different process, it doesn't share in-memory state with the main app.

The solution uses App Groups — a shared sandbox that both processes can access. When the Share Extension successfully extracts a recipe, it serializes the RecipeResponse as JSON and appends it to an array in UserDefaults(suiteName: "group.com.recipewizard.app")["pendingRecipes"].

The main RecipeWizard app observes UIApplication.willEnterForegroundNotification and calls processPendingRecipes() on every foreground event. This function reads the pending JSON, deserializes each recipe, inserts them into SwiftData via ModelContext, and clears the UserDefaults queue. It's a simple producer-consumer queue pattern using shared storage as the channel.

How would you add support for YouTube videos?

yt-dlp already supports YouTube — it was originally built for YouTube. Adding support would be:

  • Add YouTube URL patterns to TIKTOK_PATTERNS / INSTAGRAM_PATTERNS in urlValidator.ts, or create a new YOUTUBE_PATTERNS array
  • Update isSupportedUrl() to include YouTube patterns
  • Update the extractor_key detection in videoExtractor.ts to recognize "Youtube" as a platform string
  • Test that yt-dlp returns description/thumbnail for YouTube Shorts vs regular videos

The rest of the pipeline (Claude extraction, Gemini illustration, DB save) would work unchanged. The main unknown is whether YouTube caption quality is better or worse than TikTok's for recipe content.

What happens if Claude returns malformed JSON?

parseClaudeResponse() handles this in two stages. First it tries to find a ```json...``` code fence with a regex. If the fence is found, it JSON.parses the content. If the parse throws a SyntaxError, that propagates as an error — the route handler catches it and returns a 500 with the detail message.

If no fence is found, it falls back to JSON.parse(text) directly — handling the edge case where Claude outputs raw JSON without the fence. If that also fails, it throws "Claude response did not contain valid JSON".

The system prompt explicitly says "Respond ONLY with a valid JSON object wrapped in ```json ... ```" — this is the primary guardrail. In practice Claude follows this instruction very reliably with the exact wording used. The fallback handles the rare case where the instruction is partially ignored.

A more robust approach would be to use Anthropic's structured output / tool_use feature, which guarantees schema-valid JSON. That would eliminate the parsing step entirely but requires restructuring the prompt as a tool definition.

🧠 Critical Thinking Questions
What are the biggest limitations of this project?

Instagram authentication: Instagram aggressively blocks yt-dlp. The workaround is manually exporting browser cookies every ~30 days and storing them as a base64 environment variable. This is a maintenance burden and would break in production without automation.

Rate limiting is not distributed: express-rate-limit uses in-memory counters. If Railway scales to two instances or restarts the process, all counters reset. A Redis-backed rate limiter (using rate-limit-redis) would be needed for production scale.

Synchronous AI calls block the event loop: The two Claude calls and one Gemini call are sequential and each takes 1–5 seconds. For high-traffic scenarios, a job queue (BullMQ + Redis) that processes extractions asynchronously would be needed, with the iOS app polling for results.

Base64 in PostgreSQL: Storing large base64 illustrations in a text column works at small scale but has performance implications at volume. A proper solution would use cloud blob storage (S3, Vercel Blob) and store only the URL.

How would you make this production-ready for 10,000 users?

Move to async job processing: Instead of blocking the HTTP response on AI calls, enqueue the extraction as a job (BullMQ + Redis), return a job ID immediately, and let the iOS app poll a GET /jobs/:id endpoint or receive a push notification when done.

CDN for images: Upload Miyazaki illustrations to Vercel Blob or S3 on generation. Store the URL, not the base64 data. This reduces PostgreSQL row size by ~1 MB per recipe and improves query performance.

Redis-backed rate limiting: Replace in-memory rate limit storage with Redis so limits persist across restarts and multiple instances.

yt-dlp caching: Cache yt-dlp results for the same URL for ~5 minutes. Many users might share the same viral cooking video simultaneously.

Horizontal scaling: All state is in PostgreSQL (stateless app tier) so Railway can scale to multiple instances freely, except for the in-memory rate limiter (solved by Redis).

Instagram cookies automation: Automate cookie refresh via a scheduled job using a headless browser or Instagram's unofficial API.

What would you change if you could rebuild it?

Use Anthropic's tool_use feature instead of manual JSON parsing. Instead of prompting Claude to return a ```json``` block, I'd define the recipe schema as a tool definition and force the model to call it. This guarantees valid structured output and eliminates the parseClaudeResponse() parsing step entirely.

Add a job queue from day one. The synchronous AI pipeline is the biggest architectural constraint. A job queue would allow better error handling, retries, progress reporting, and horizontal scaling — all of which I'd want before going to production.

Add prompt caching. The SYSTEM_PROMPT in promptBuilder.ts is the same for every request. Anthropic's prompt caching feature would cache the system prompt's token computation, reducing extraction cost by ~90% on the system prompt tokens for the second+ call in a 5-minute window.

Add tests for the AI integration. There are unit tests for urlValidator and claudeService, but no integration tests that mock the Claude API and verify the full pipeline end-to-end. I'd add a set of fixture-based tests that feed known video metadata and assert on the extraction quality.

How would you test this project?

The project uses Vitest for the backend. Current test coverage:

  • urlValidator.test.ts — tests isSupportedUrl() and extractUrlFromText() with various URL formats
  • claudeService.test.ts — tests parseClaudeResponse() with different JSON formats (fenced, raw, malformed)
  • recipes.test.ts — integration tests for the POST /extract route using supertest

What I'd add: (1) Tests for videoExtractor that mock the yt-dlp subprocess output; (2) End-to-end extraction tests with a fixed response from a mocked Anthropic client, verifying that the normalization logic (nullish coalescing, confidence clamping, sortOrder assignment) works correctly; (3) Auth middleware tests for expired tokens, missing headers, and invalid signatures.

For iOS, the Share Extension logic is the hardest to test — it requires a physical device and a configured App Group. I'd add unit tests for the URL extraction regex logic in ShareViewController using XCTest.

How is this app different from just asking ChatGPT for a recipe?

There are three key differences. First, source authenticity — RecipeWizard extracts the actual recipe from the video you watched, with the exact measurements and techniques that specific creator used. Asking ChatGPT for "garlic butter pasta" gives you a generic recipe that might be totally different from the one in the video you loved.

Second, structured persistence — the recipe is saved with proper structure: ingredients as a database table, steps as an ordered list, tags for filtering. It's searchable, browsable, and integrated with iOS's native UI. A ChatGPT conversation is ephemeral.

Third, zero copy-paste — the entire flow is one tap: share the video from TikTok, the recipe appears in your app. No manual transcription, no copy-pasting the caption, no reformatting. The extraction is automatic from the share sheet.

🎯 Code Gotchas

Specific tricky lines an interviewer might point to — and how to explain them:

const PORT = parseInt(process.env.PORT ?? "8000", 10);
What does the 10 do? Why not just Number(process.env.PORT)?
parseInt(str, radix) — the second argument is the radix (base). Without it, some JS engines interpret strings starting with "0" as octal (base 8). With radix 10, "0100" → 100 not 64. Number("8000") would also work here, but parseInt is more explicit and handles strings with trailing non-numeric chars gracefully (e.g. "8000px" → 8000).
if (require.main === module) { app.listen(...); }
What does this condition check?
require.main is the module that Node directly executed. module is the current file's module object. When you run node dist/index.js, they're equal → server starts. When the test suite imports app (e.g. import { app } from "../index"), they're different → server doesn't auto-start. This pattern lets you export the app for testing with supertest without accidentally starting the server.
const match = text.match(/```json\s*([\s\S]*?)```/);
Why [\s\S]* instead of .* ? What does the ? do?
In JavaScript, . doesn't match newline characters by default. Since Claude's JSON response spans multiple lines, .* would only match within a single line and fail. [\s\S] means "whitespace OR non-whitespace" — which matches every character including newlines. The *? makes it lazy — it stops at the FIRST ``` it finds. Without the ? (greedy), it would match everything between the first opening fence and the LAST closing fence in the entire text, which would be wrong if there are multiple code blocks.
const confidence = typeof extracted.confidence === "number" ? Math.max(0, Math.min(1, extracted.confidence)) : 0.5;
Why not just use extracted.confidence directly?
Two reasons: (1) Claude is instructed to return a float 0.0–1.0 but LLMs can hallucinate values like 0.95 that would be fine, or values like 1.5 or -0.2 that would break the PostgreSQL Float constraint or iOS display logic. Math.max(0, Math.min(1, x)) clamps any value to [0,1]. (2) The typeof check handles the case where Claude omits the confidence field entirely or returns it as a string — without it, Math.max(0, Math.min(1, undefined)) would return NaN which would fail the DB write.
args.unshift("--cookies", resolvedCookiesPath);
Why unshift instead of push?
push() adds to the end. unshift() adds to the beginning. The yt-dlp argument array ends with the URL: ["--dump-json", "--no-playlist", "--no-warnings", "https://..."]. The URL must always be the LAST argument. --cookies /path/file.txt must come BEFORE the URL or yt-dlp may not parse it correctly. unshift prepends the two cookie arguments to the front, keeping the URL at the end.
saveRecipeForUser(req.user!.userId, responsePayload).catch(err => { console.error("[recipes] Failed to save recipe to DB:", err); });
Why is there no await? What's the ! for?
No await is intentional — fire-and-forget pattern. The DB save happens asynchronously after the response is sent, so the user doesn't wait for the DB write. The .catch() prevents an unhandled promise rejection (which would crash the process in Node 15+). The ! (non-null assertion operator) tells TypeScript "I know req.user is not undefined here" — this is safe because requireAuth middleware on line 13 guarantees req.user is set before this handler runs.
const globalForPrisma = globalThis as unknown as { prisma: PrismaClient }; export const prisma = globalForPrisma.prisma ?? new PrismaClient(); if (process.env.NODE_ENV !== "production") globalForPrisma.prisma = prisma;
Why is the Prisma client stored on globalThis?
In development with hot-reload (ts-node-dev), every file edit causes the module to re-evaluate. Without this pattern, a new PrismaClient would be created on every reload, quickly exhausting PostgreSQL's connection pool. By storing the client on globalThis (which persists across module re-evaluations), the same client is reused across hot-reloads. In production, this never applies because the process doesn't reload — the !== "production" guard skips the globalThis write to avoid confusion.
