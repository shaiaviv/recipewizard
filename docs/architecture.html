Architecture & Data Flow

How RecipeWizard is structured, why each piece exists, and the exact path data travels from iOS tap to database row.

🎯 Architecture Overview

RecipeWizard solves a specific problem: social media cooking videos are rich in information but that information is locked in a non-machine-readable format — video frames + free-text captions. The architecture is a three-layer extraction pipeline:

  1. Video layer (yt-dlp): reads the video's metadata without downloading any video bytes — title, caption, thumbnail URL.
  2. AI layer (Claude): converts unstructured text + image → structured JSON recipe. Two-stage: cheap classifier first, expensive extractor second.
  3. Persistence layer (Prisma + PostgreSQL): stores recipes per-user so they survive app reinstalls and device changes.

The iOS app is a thin client — it provides the UI, handles OAuth sign-in, and renders what the backend returns. The Share Extension is a separate iOS process that writes to a shared UserDefaults queue; the main app drains it on foreground.

🔧 Technology Decisions
Library / ToolWhat it doesWhy chosenWhat breaks without it
yt-dlp CLI tool that extracts video metadata from 1000+ sites including TikTok and Instagram Only tool that reliably handles TikTok and Instagram authentication, short links, and metadata extraction without downloading video. Actively maintained (vs. youtube-dl) No way to get video titles/descriptions/thumbnails for social platforms without scraping HTML
@anthropic-ai/sdk Official Anthropic SDK for Claude API (messages.create) Claude is the only model family with both excellent multimodal understanding of food images and structured JSON output following complex schemas Recipe extraction — the core feature
@google/genai Google's Generative AI SDK for Gemini Gemini 2.5 Flash has native image generation (IMAGE response modality). Claude doesn't generate images. DALL-E 3 wasn't chosen because Gemini produces the consistent watercolor/anime aesthetic better. Illustrations fall back to original thumbnail — still functional
sharp C++ image processing (libvips) — resize, format conversion ~100x faster than pure-JS image libs. Handles WebP, HEIC, PNG → JPEG in memory. Critical because image processing happens on every request. Thumbnail too large for Claude's API (Anthropic has a ~5MB image limit)
Prisma ORM Type-safe ORM with auto-generated TypeScript client from schema.prisma Generates TypeScript types from schema — no type drift between DB and code. Nested writes handle relational data elegantly. Migrations are tracked and versioned. Raw SQL queries with no type safety; manual relation management
express-rate-limit In-memory request counting per key Simple, no Redis required for single-instance Railway deployment. Per-user keying prevents API abuse. API credits could be exhausted by a single abusive user
google-auth-library Official Google library for verifying ID tokens Handles public key fetching from Google's JWKS endpoint, token signature verification (RS256), expiry checking. DIY JWT verification would be a security mistake. Authentication — the only auth mechanism
SwiftData (iOS) Apple's persistence framework (Core Data successor) Native iOS 17+, integrates directly with SwiftUI (@Query). Automatic CloudKit sync path. Handles offline access. Recipes only visible while online
🎨 Design Patterns Used
Classifier Gating
A cheap binary classifier (Haiku, 5 tokens) gates an expensive generator (Sonnet, 4096 tokens). Applied to skip non-recipe videos. Reduces AI cost by ~50% given typical share behavior.
Pipeline Pattern
4 sequential stages in routes/recipes.ts, each with independent error handling and HTTP status codes. Failure in any stage short-circuits the pipeline.
Module Augmentation (TypeScript)
declare global { namespace Express { interface Request { user? } } } — extends a third-party type without forking the library. Enables req.user type safety across all route files.
Upsert (Idempotent Write)
Prisma's upsert for user creation — atomically "find or create". Handles concurrent sign-ins from multiple devices without race conditions.
Fire-and-Forget
DB save called without await, with only a .catch() for logging. Decouples response latency from DB write latency — the user gets their recipe instantly.
Actor Concurrency (Swift)
iOS RecipeAPIService is declared as actor — guarantees serial access to internal state. All URLSession calls are isolated, preventing data races in the Share Extension's concurrent environment.
UserDefaults Queue (iOS IPC)
Share Extension cannot access SwiftData (memory limit, separate process). Writes [RecipeResponse] JSON to UserDefaults(suiteName: appGroupID). Main app drains this queue on foreground — a simple IPC via shared storage.
Cascade Delete (Prisma)
Prisma schema: onDelete: Cascade on Ingredient and Step relations. Deleting a Recipe automatically deletes all its ingredients and steps atomically. No manual cleanup needed.
⚠️ Known Limitations
Rate limiting is in-memory only
express-rate-limit stores counters in memory. On Railway, if the process restarts or scales to multiple instances, counters reset. For true rate limiting at scale, Redis-backed storage (express-rate-limit's redisStore option) would be needed.
Instagram requires manual cookie refresh
Instagram blocks yt-dlp requests without authenticated cookies. Cookies expire approximately every 30 days and must be manually re-exported from a logged-in browser session. The health endpoint tracks cookie age and warns when they're > 30 days old.
No retry logic for yt-dlp or Claude
If yt-dlp returns a transient error (network blip, rate limit from TikTok), the request fails immediately. No exponential backoff or retry. For production, wrapping yt-dlp calls in a retry loop with backoff would improve reliability.
Thumbnail base64 stored in PostgreSQL
The Miyazaki illustration (up to ~1–2 MB base64) is stored in a PostgreSQL text column. For high volume, a blob storage service (S3/Vercel Blob) would be more appropriate. PostgreSQL handles it fine at small scale.
CloudKit sync disabled
SwiftData CloudKit sync is commented out — requires a paid Apple Developer account and CloudKit entitlements. Recipes are stored locally in SwiftData but not synced across devices. The backend API fills this gap (fetchUserRecipes syncs on foreground).
Simulator testing not possible for Share Extension
App Groups and Share Extensions don't work correctly in the iOS Simulator — requires a physical device with a valid provisioning profile. Also limits automated testing of the extension.
